# ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## PCF (Pentest Collaboration Framework)

---

# 1. Общие сведения

## 1.1 Наименование системы

Pentest Collaboration Framework (PCF)

## 1.2 Назначение системы

Система предназначена для обеспечения централизованного хранения, обработки и управления данными пентест-проектов, а также организации совместной работы специалистов по информационной безопасности.

---

## 1.3 Цель разработки

Целью разработки является создание программного продукта, обеспечивающего:

- структурированное хранение данных по пентесту
- совместную работу команды
- фиксацию и сопровождение уязвимостей
- хранение доказательной базы
- формирование отчётной документации

---

# 2. Область применения

Система применяется:

- в командах пентестеров
- при проведении внутренних и внешних тестов на проникновение
- при подготовке отчётов по результатам тестирования безопасности

---

# 3. Термины и определения


| Термин         | Определение                                        |
| -------------- | -------------------------------------------------- |
| Проект         | Совокупность данных, относящихся к одному пентесту |
| Хост           | Узел инфраструктуры (IP-адрес или домен)           |
| Endpoint       | Точка взаимодействия (например, HTTP-ручка)        |
| Уязвимость     | Найденная проблема безопасности                    |
| Доказательство | Материалы, подтверждающие уязвимость               |
| Участник       | Пользователь системы, имеющий доступ к проекту     |


---

# 4. Общая характеристика системы

## 4.1 Описание системы

Система представляет собой веб-приложение, обеспечивающее:

- управление проектами
- хранение информации об инфраструктуре цели
- фиксацию уязвимостей
- ведение обсуждений
- формирование отчётов

---

## 4.2 Основные функции

Система должна обеспечивать выполнение следующих функций:

- управление пользователями и ролями
- управление проектами
- управление активами (hosts, ports, services, endpoints)
- управление уязвимостями
- загрузка и хранение файлов
- импорт данных
- ведение комментариев
- формирование отчётов
- ведение журнала действий

---

## 4.3 Технологический стек


| Компонент               | Технология                         |
| ----------------------- | ---------------------------------- |
| Backend                 | Python (REST API, префикс /api/v1/)|
| Frontend                | React + TypeScript                 |
| СУБД                    | PostgreSQL                         |
| Файловое хранилище      | MinIO                              |
| Real-time синхронизация | WebSocket                          |
| Аутентификация          | JWT в httpOnly cookie (access + refresh) |
| Контейнеризация         | Docker Compose                     |
| Пагинация               | Offset (?page=1&size=20)           |


---

# 5. Требования к функциональности

---

## 5.1 Управление пользователями

Система является **закрытой**: самостоятельная регистрация пользователей не предусмотрена. Аккаунты создаются только администратором.

Система должна обеспечивать:

- создание учётных записей пользователей администратором
- аутентификацию пользователей
- разграничение прав доступа на основе ролей

В системе предусмотрены следующие роли:


| Роль          | Права доступа                                                                                              |
| ------------- | ---------------------------------------------------------------------------------------------------------- |
| Администратор | Управление пользователями; полный доступ ко всем проектам; просмотр журнала действий; назначение ролей     |
| Пентестер     | Работа в рамках назначенных проектов: управление активами, уязвимостями, файлами, комментариями и отчётами |


---

## 5.2 Управление проектами

Система должна обеспечивать:

- создание проекта
- редактирование проекта
- удаление проекта
- просмотр списка проектов
- управление участниками проекта
- назначение ролей участникам

---

## 5.3 Управление активами

Система должна обеспечивать ведение структуры активов:

- добавление, редактирование и удаление хостов
- добавление, редактирование и удаление портов
- добавление, редактирование и удаление сервисов
- добавление, редактирование и удаление endpoints

---

## 5.4 Управление уязвимостями

Система должна обеспечивать:

- создание уязвимости
- редактирование уязвимости
- удаление уязвимости
- просмотр уязвимостей

---

Для каждой уязвимости должна быть предусмотрена возможность указания:

- наименования
- описания
- уровня критичности
- CVSS: пользователь выбирает версию (v3.1 или v4.0) и вводит числовой score и векторную строку вручную
- CWE: свободный текст (например, «CWE-89»)
- статуса
- шагов воспроизведения
- влияния
- рекомендаций

---

## 5.5 Связи между сущностями

Система должна обеспечивать:

- обязательную привязку каждой уязвимости к конкретному хосту
- переход между хостом и привязанными к нему уязвимостями

---

## 5.6 Управление файлами

Файлы (доказательная база) хранятся в объектном хранилище **MinIO**.

Система должна обеспечивать:

- загрузку файлов (скриншоты, текстовые файлы, архивы) при описании уязвимости
- привязку одного или нескольких файлов к уязвимости
- просмотр загруженных файлов
- удаление файлов

Поддерживаемые типы файлов: PNG, JPG, GIF, WebP (изображения); TXT, PDF, XML, JSON (документы); ZIP, TAR.GZ (архивы).

Максимальный размер одного файла: **50 МБ**.

---

## 5.7 Комментарии

Система должна обеспечивать:

- добавление комментариев к уязвимостям
- просмотр комментариев
- редактирование собственного комментария
- удаление собственного комментария
- упоминание пользователей через @username
- отправку in-app уведомления упомянутому пользователю (значок/бейдж в интерфейсе)

---

## 5.8 Импорт данных

Система должна обеспечивать:

- загрузку данных в формате JSON
- обработку загруженных данных
- добавление данных в структуру проекта

Формат JSON-файла — **собственный формат PCF** (не Nmap, не Burp Suite). Схема включает объекты: хосты, порты, сервисы, endpoints. Схема фиксируется в документации API и является обязательной для валидации при загрузке.

---

## 5.9 Формирование отчётов

Система должна обеспечивать:

- формирование отчёта по проекту
- выгрузку отчёта в форматах: **Markdown, PDF, DOCX**

Отчёт должен включать следующие разделы:

- общая информация о проекте (название, даты, участники)
- сводная статистика уязвимостей по критичности и статусу
- сводная таблица активов (хосты, порты)
- список уязвимостей с полным описанием (название, критичность, CVSS, CWE, статус, шаги воспроизведения, влияние, рекомендации)
- скриншоты (встроенные в документ или прикреплённые)

---

## 5.10 Журнал действий

Система должна обеспечивать:

- фиксацию действий пользователей
- доступ к журналу только для пользователей с соответствующими правами

---

## 5.11 Совместная работа

Система должна обеспечивать:

- одновременную работу нескольких пользователей
- актуальность отображаемых данных
- синхронизацию изменений в режиме реального времени посредством **WebSocket**
- автоматическое обновление интерфейса при изменении данных другим пользователем без перезагрузки страницы
- рассылку WebSocket-событий при **любых CRUD-операциях** над сущностями проекта: хостами, портами, сервисами, endpoints, уязвимостями, комментариями, файлами

---

# 6. Требования к пользовательскому интерфейсу

Система должна обеспечивать:

- интуитивно понятный интерфейс
- логичную навигацию между сущностями
- быстрый доступ к связанным данным
- удобный ввод и редактирование информации
- левую навигационную панель (sidebar) в стиле wiki/Confluence для работы со структурой проекта
- древовидную иерархию в интерфейсе: **Проект → Хосты → (Порты, Endpoints)** с переходом по узлам без перезагрузки страницы
- отображение уязвимостей только в контексте выбранного хоста
- гибридный режим sidebar: панель видна по умолчанию и может сворачиваться пользователем
- сохранение контекста выбранного узла (активный проект/хост/раздел) при обновлении данных в real-time

---

# 7. Требования к надежности

Система должна обеспечивать:

- сохранность данных пользователей
- корректную обработку ошибок
- устойчивость к некорректному вводу данных

---

# 8. Требования к безопасности

Система должна обеспечивать:

- аутентификацию пользователей на основе **JWT-токенов** по схеме access + refresh: оба токена передаются и хранятся исключительно в **httpOnly cookie** (не доступны из JavaScript); access-токен — короткоживущий, refresh-токен — долгоживущий
- защиту от CSRF-атак: cookie выставляется с атрибутом `SameSite=Strict`; для запросов, изменяющих состояние (POST/PUT/PATCH/DELETE), проверяется заголовок `Origin`
- хранение паролей с использованием хэширования (**bcrypt**)
- разграничение прав доступа на основе ролей
- защиту данных от несанкционированного доступа
- валидацию и санитизацию входных данных на стороне сервера

---

# 9. Требования к эксплуатации

Система должна обеспечивать:

- развертывание с помощью **Docker Compose**
- возможность одновременной работы нескольких пользователей

Состав контейнеров Docker Compose:


| Контейнер | Назначение                         |
| --------- | ---------------------------------- |
| backend   | Python REST API + WebSocket сервер |
| frontend  | React + TypeScript SPA             |
| db        | PostgreSQL — основная СУБД         |
| minio     | MinIO — объектное хранилище файлов |


---

# 10. Требования к результатам работы системы

В результате работы системы пользователь должен иметь возможность:

- вести полный цикл пентест-проекта
- хранить всю информацию в одном месте
- совместно работать с командой
- формировать готовый отчёт

---

# 11. Критерии приемки

Система считается соответствующей требованиям, если:

- реализованы все заявленные функции
- пользователь может выполнить основные сценарии работы
- данные корректно сохраняются и отображаются
- отчёт формируется без ошибок
